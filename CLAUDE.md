# 育成シミュレーションゲーム

## プロジェクト概要

女の子育成シミュレーションゲーム（オフライン）。プリンセスメーカー / ときメモ系統。

**ステータス**: 企画段階

## 技術スタック

| カテゴリ | 技術 |
|---|---|
| フレームワーク | React + TypeScript |
| ゲームエンジン | Phaser 3（2Dゲーム向け軽量エンジン） |
| UI | React（メニュー・ダイアログ）+ Phaser（ゲーム画面） |
| データ永続化 | localStorage / IndexedDB |
| ビルド | Vite |
| 配布 | Electron（デスクトップアプリ化）or Web |

## ゲームシステム（仮）

### コアループ
1. 日付が進む（1日 or 1週間単位）
2. スケジュールを組む（勉強・運動・バイト・デート等）
3. ステータスが変動する
4. イベントが発生する
5. エンディング分岐

### ステータス例
- 知力、体力、魅力、芸術、料理、愛情度
- 疲労、ストレス、所持金

### イベント
- 季節イベント（花見、海、クリスマス等）
- 好感度イベント（デート、告白）
- ランダムイベント

## 開発コマンド

```bash
# 依存関係インストール
npm install

# 開発サーバー起動
npm run dev

# ビルド
npm run build

# Electron で実行（デスクトップアプリ）
npm run electron
```

## ディレクトリ構成（予定）

```
src/
  components/     # React UI コンポーネント
  game/           # Phaser ゲームロジック
    scenes/       # Phaser シーン
    entities/     # キャラクター・オブジェクト
  data/           # ゲームデータ（イベント、セリフ等）
  hooks/          # React カスタムフック
  stores/         # 状態管理（Zustand）
  types/          # TypeScript 型定義
  assets/         # 画像・音声
public/
  images/         # キャラクター立ち絵・背景
  audio/          # BGM・SE
```

## AI アシスタントへの指示

- 仕様や要件が曖昧な場合は、推測で進めず **AskUserQuestion ツール** や **/interview** を使ってユーザーに確認すること
- ゲームバランス（ステータス計算式・イベント発生条件）に関わる変更は、影響範囲を明示して確認を取ること
- 複数の解釈が可能な指示を受けた場合は、選択肢を提示して判断を仰ぐこと

## コンテンツガイドライン

- 恋愛・ロマンス要素: OK
- ほのかなお色気（水着・温泉イベント等）: OK（示唆程度）
- 露骨な性的描写: NG（Claude Code ポリシー外）

## ワークフロー

1. **プラン作成**: 実装前にチェック可能な項目リストを作る
2. **プラン確認**: 着手前にユーザーとすり合わせる
3. **進捗追跡**: 完了した項目は即座にマークする
4. **変更説明**: 各ステップでハイレベルなサマリーを共有する
5. **結果記録**: レビュー結果をプランに追記する

## コア原則

- **シンプルさ優先**: 変更はできるだけ単純に。影響範囲を最小限にする
- **根本原因を探す**: 一時しのぎではなく根本解決
- **最小影響**: 必要な箇所だけ触る。バグを持ち込まない
